# AGENTS.md

本文件供 Codex 和其他开发代理使用，记录 LiveBoard 的工作约定、运行方式、设计决策和容易重复出现的问题。对项目进行修改前应先阅读本文件；完成具有长期影响的改动后同步更新相关章节。

## 项目目标

LiveBoard 是自托管教学工作台。产品设计应优先服务真实教学任务：整理课程内容、授课、练习反馈、权限分配和课程交流。避免为了展示技术而增加低频功能或装饰性界面。

## 代码结构

- `apps/web`：Next.js App Router 前端。
- `apps/api`：NestJS API 与 Prisma 数据访问。
- `packages/shared`：前后端共享类型、权限和评分纯函数。
- `apps/web/lib/routes.ts`：应用路由唯一来源，组件不要手写内部路径。
- `apps/web/lib/api`：前端 API 调用唯一入口，页面组件不要直接调用 `fetch`。
- API 按 `apps/api/src/modules/<domain>` 划分业务域。

## 必须遵守的开发流程

1. 修改前先定位现有实现、共享类型和相关样式，不凭页面截图猜测数据结构。
2. 保留用户已有改动；不要重置、覆盖或清理不属于当前任务的内容。
3. 文件编辑使用补丁方式，避免整文件机械覆盖造成无关差异。
4. 修改共享类型后先构建 `packages/shared`，开发中的 API watcher 可能暂时读取旧的 `dist` 类型。
5. 完成后至少运行相关项目的 typecheck；功能或跨层修改运行 `pnpm validate`。
6. 涉及 UI 时同时检查常规桌面宽度和窄屏，不以“构建通过”代替视觉验证。
7. 不提交 `.env`、密钥、用户数据、构建产物或依赖目录。

## 开发与生产服务

开发模式：

```bash
pnpm infra:up
pnpm dev
```

- Web：Next.js dev server，端口 3000，支持热更新。
- API：NestJS watch mode，端口 4000。
- PostgreSQL、Redis、MinIO：Docker 容器。

生产容器：

```bash
docker compose up --build -d
```

正式生产部署只使用 GitHub Release 单文件包。服务器不拉取源码、不运行 pnpm、不构建应用镜像，也不直接下载容器镜像；用户在电脑下载 `liveboard-<version>-linux-amd64.tar.gz` 后上传服务器并运行包内 `deploy.sh`。脚本必须在迁移前备份 PostgreSQL，并等待 API 与 Web 都健康后才报告完成。不得使用 `docker compose down -v` 或清理命名卷更新服务。MinIO 需要独立备份或卷快照。

生产首次初始化由 `apps/api/src/bootstrap-production.ts` 自动完成，只创建一个随机密码的最高管理员、默认 workspace 和论坛分类。部署脚本必须在最终总结中醒目显示首次凭据，并将其保存到权限为 `600` 的 `/opt/liveboard/initial-admin-credentials.txt`；用户修改密码后应删除该文件。升级不得生成或覆盖管理员密码。`apps/api/prisma/seed.cjs` 仅用于本地演示，生产部署不得调用。

`docker-compose.yml` 中的 Web 明确使用 `NODE_ENV=production`。用户要求“开发版本”时，不得只执行 `docker compose up`；应停止 `web`、`api` 容器，保留基础设施，再运行 `pnpm dev`。

在 Codex 沙箱中，本机端口或数据库连接可能返回 `EPERM`/`P1001`。确认基础设施正常后，应在获准的非沙箱环境启动开发进程，不要误判为项目故障。

## UI 设计原则

- 参考 Notion、Claude、Reddit 等产品的信息密度与交互克制，但不复制品牌外观。
- 优先扁平信息层级、清晰间距和连续列表；减少嵌套卡片、色块、阴影和无意义圆角。
- 页面主任务必须显眼，低频操作放入三点菜单或次级区域。
- 按钮文字使用具体动作，例如“发布主题”“添加成员”“授权”，避免模糊的“确认”。
- 不写营销式、AI 式副标题。说明文字应直接描述用户能做什么。
- 空状态应紧凑且提供下一步，不得用大面积空白撑高页面。
- 危险操作与常规操作分开；不可执行的操作优先隐藏，只有需要解释时才禁用。
- 操作成功、警告和错误反馈统一使用页面顶部的全局悬浮通知，不在内容区插入反馈色条；表单字段自身的校验说明除外。

## CSS 易错点

项目历史上 `apps/web/app/globals.css` 与 `apps/web/app/redesign.css` 存在多轮同名选择器。新增路由级改版时：

- 优先创建路由专属 CSS，并从该路由的 `layout.tsx` 引入。
- 明确重置继承的 `display`、`grid-template-columns`、`align-items`、`overflow`、`min-width`、高度和最后一项样式。
- 不要只覆盖颜色、边框和圆角；旧网格规则仍可能把输入框或列表项拉伸。
- 对 Grid/Flex 子项设置 `min-width: 0`，防止文本和表单横向撑破容器。
- 表单从双列改为侧栏单列时，必须显式写 `grid-template-columns: minmax(0, 1fr)`。
- 修改后检查 `:first-child`、`:last-child` 等旧选择器是否因 DOM 结构变化命中了错误元素。

曾经出现并已修复的问题：

- 权限组成员区继承旧 `display: grid` 后自动拉伸，造成搜索框巨大和列表被截断。
- 成员头像插入第一列后，旧的 `> span:first-child` 命中头像，姓名和用户名失去纵向布局。
- 权限组右侧 280px 侧栏继承双列表单规则，导致“用途说明”被挤成竖排并横向溢出。
- 论坛全局多层覆盖产生巨大空白和圆角残留，后改为路由独立样式。

## 内容与权限

- 文件夹树接口返回 `fileCount` 和 `children`。只有两者都为空时，前端才显示“删除空文件夹”。后端仍是最终删除校验边界。
- 文件夹和文件的权限入口位于各自三点菜单；文件编辑页的权限入口也位于更多菜单。
- 权限弹窗必须携带明确的 `folder` 或 `file` 目标，不能隐式使用当前选中文件夹管理文件权限。
- 权限级别为 `owner`、`editor`、`lecturer`、`viewer`、`no_access`。
- 权限按钮的显示与隐藏只是体验优化，服务层必须检查有效权限。

## 已确认的产品决策

- 导航名称使用“论坛”，不使用“讨论”。
- 导航和页面名称使用“内容”，不使用“课程内容”。
- 创建在线测验不包含“选择测验范围”。
- 测验选项逐行添加，不让用户在大文本框中手动换行分隔。
- 课程演示根据一级、二级标题拆分页，支持方向键、空格、PageUp/PageDown、Home 和 End。
- “授课”是所有已登录用户可访问的独立板块；课件由内容段落快照与实时练习引用拼装，创建者和系统管理员可编辑。内容文件编辑页不再提供直接授课入口。
- 登录页的演示账号由 `NEXT_PUBLIC_SHOW_DEMO_ACCOUNTS` 控制；该变量写入 Web 构建产物，生产容器修改后需要重新 build。
- AI 设置支持保存多个命名模型配置并显式切换当前配置；配置使用常见 OpenAI 兼容服务商预设，只有“其他兼容服务”允许填写 API 地址。回答温度由服务端固定为 `0.2`，不作为管理员可调参数；当前配置不能直接删除，必须先切换。

## 数据与安全

- 系统角色为 `super_admin`、`admin`、`member`；普通用户的内容能力由权限组和资源授权决定。
- 后端服务层是权限安全边界。
- 所有 schema 变更都应提交 Prisma migration，不使用 `db push`；测试数据库需要重建时使用 `pnpm db:reset`。
- Release 部署必须自动生成 `SESSION_SECRET`、数据库密码和 MinIO 凭据。HTTP IP 模式使用 `SESSION_COOKIE_SECURE=false`；配置 HTTPS 后改为 `true`。

## 验证清单

提交前按风险选择并记录：

```bash
pnpm format:check
pnpm typecheck
pnpm test
pnpm build
# 或一次执行
pnpm validate
```

UI 修改额外确认：

- 页面没有横向滚动或截断。
- 输入框、下拉框和列表项没有被父级 Grid 拉伸。
- 弹窗在小屏内可滚动且操作按钮可达。
- 空、有少量数据、有大量数据三种状态均合理。
- 三点菜单在视口边缘不会溢出。
