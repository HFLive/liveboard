# AGENTS.md

本文件供 Codex 和其他开发代理使用，记录 LiveBoard 的工作约定、运行方式、设计决策和容易重复出现的问题。对项目进行修改前应先阅读本文件；完成具有长期影响的改动后同步更新相关章节。

## 项目目标

LiveBoard 是自托管教学工作台。产品设计应优先服务真实教学任务：整理课程内容、授课、练习反馈、权限分配和课程交流。避免为了展示技术而增加低频功能或装饰性界面。

## 代码结构

- `apps/web`：Next.js App Router 前端。
- `apps/api`：NestJS API 与 Prisma 数据访问。
- `packages/shared`：前后端共享类型、权限和评分纯函数。
- `apps/web/lib/routes.ts`：应用路由唯一来源，组件不要手写内部路径。
- `apps/web/lib/api`：前端 API 调用唯一入口，页面组件不要直接调用 `fetch`。
- API 按 `apps/api/src/modules/<domain>` 划分业务域。

## 必须遵守的开发流程

1. 修改前先定位现有实现、共享类型和相关样式，不凭页面截图猜测数据结构。
2. 保留用户已有改动；不要重置、覆盖或清理不属于当前任务的内容。
3. 文件编辑使用补丁方式，避免整文件机械覆盖造成无关差异。
4. 修改共享类型后先构建 `packages/shared`，开发中的 API watcher 可能暂时读取旧的 `dist` 类型。
5. 完成后至少运行相关项目的 typecheck；功能或跨层修改运行 `pnpm validate`。
6. 涉及 UI 时同时检查常规桌面宽度和窄屏，不以“构建通过”代替视觉验证。
7. 不提交 `.env`、密钥、用户数据、构建产物或依赖目录。
8. `apps/web/next-env.d.ts` 由 Next.js 自动生成且不纳入版本控制；Web typecheck 必须先运行 `next typegen`，不要手动修改或提交该文件。

## 开发与生产服务

开发模式：

```bash
pnpm infra:up
pnpm dev
```

- Web：Next.js dev server，端口 3000，支持热更新。
- API：NestJS watch mode，端口 4000。
- PostgreSQL、Redis、MinIO：Docker 容器。

生产容器：

```bash
docker compose up --build -d
```

正式生产部署只使用 GitHub Release 单文件包。服务器不拉取源码、不运行 pnpm、不构建应用镜像，也不直接下载容器镜像；用户在电脑下载 `liveboard-<version>-linux-amd64.tar.gz` 后上传服务器并运行包内 `deploy.sh`。脚本必须在迁移前备份 PostgreSQL，并等待 API 与 Web 都健康后才报告完成。不得使用 `docker compose down -v` 或清理命名卷更新服务。MinIO 需要独立备份或卷快照。

生产首次初始化由 `apps/api/src/bootstrap-production.ts` 自动完成，只创建一个随机密码的最高管理员、默认 workspace 和论坛分类。部署脚本必须在最终总结中醒目显示首次凭据，并将其保存到权限为 `600` 的 `/opt/liveboard/initial-admin-credentials.txt`；用户修改密码后应删除该文件。升级不得生成或覆盖管理员密码。`apps/api/prisma/seed.cjs` 仅用于本地演示，生产部署不得调用。

`docker-compose.yml` 中的 Web 明确使用 `NODE_ENV=production`。用户要求“开发版本”时，不得只执行 `docker compose up`；应停止 `web`、`api` 容器，保留基础设施，再运行 `pnpm dev`。

在 Codex 沙箱中，本机端口或数据库连接可能返回 `EPERM`/`P1001`。确认基础设施正常后，应在获准的非沙箱环境启动开发进程，不要误判为项目故障。

## UI 设计原则

- 参考 Notion、Claude、Reddit 等产品的信息密度与交互克制，但不复制品牌外观。
- 优先扁平信息层级、清晰间距和连续列表；减少嵌套卡片、色块、阴影和无意义圆角。
- 页面主任务必须显眼，低频操作放入三点菜单或次级区域。
- 按钮文字使用具体动作，例如“发布主题”“添加成员”“授权”，避免模糊的“确认”。
- 不写营销式、AI 式副标题。说明文字应直接描述用户能做什么。
- 空状态应紧凑且提供下一步，不得用大面积空白撑高页面。
- 危险操作与常规操作分开；不可执行的操作优先隐藏，只有需要解释时才禁用。
- 操作成功、警告和错误反馈统一使用页面顶部的全局悬浮通知，不在内容区插入反馈色条；表单字段自身的校验说明除外。
- 应用页面不显示独立的页面级大标题区，使用导航选中态、面板标题和内容自身标题提供上下文；详情页仍保留必要的返回或操作按钮，文档编辑器的可编辑文件名不受此规则影响。
- 应用页面（含 AI 对话页）的工作区组件统一缩小约 30%，不要在单个路由中重新放大按钮、表单或面板破坏整体密度。
- 高密度工作区只保留主操作使用深色实心按钮；次要操作使用浅底无边框样式，危险操作使用浅红底，表格和行内操作使用无底色文字或图标。布局优先连续平面、分隔线和留白，避免密集页面出现大量描边按钮与嵌套卡片。
- 配色统一使用 `redesign.css :root` 的语义变量，禁止再写近似色字面量：画布 `--bg`；填充层级 `--fill-subtle`（输入框/信息块）、`--fill-hover`（唯一列表 hover 灰）、`--fill-active`（选中/按下的中性底）；线条 `--line`（弱分隔）、`--line-strong`（结构线）；文字 `--text`/`--text-soft`/`--muted`；强调 `--accent`/`--accent-soft`/`--accent-dark`。黄色只用于选中状态、强调图标和必要提示。
- 页面级区块不再使用圆角矩形外框（无 1px 外框、无面板圆角）；结构层级由分隔线、双栏布局的竖线和留白提供。列表与工作面板使用透明背景，不使用白色大卡片。圆角只保留给控件和小元素：常规控件 10px、标签与小块 5-8px、弹窗 18px。
- 面板标题区是纯文字标题加底部通长分隔线；表格表头为透明底文字加分隔线；文档页等追求极简的页面可以省略标题区横线和分栏竖线，直接用留白分区。
- 状态筛选和紧凑视图切换统一使用论坛式分段控件：不显示整块底板或外框，选项平铺，仅为当前项提供浅色圆角底。
- 空表格行、空列表和空面板不是可交互列表项，不得响应列表 hover；空单元格背景始终与页面背景一致，禁止写死为白色。

## CSS 易错点

项目历史上 `apps/web/app/globals.css` 与 `apps/web/app/redesign.css` 存在多轮同名选择器。新增路由级改版时：

- 优先创建路由专属 CSS，并从该路由的 `layout.tsx` 引入。
- 明确重置继承的 `display`、`grid-template-columns`、`align-items`、`overflow`、`min-width`、高度和最后一项样式。
- 不要只覆盖颜色、边框和圆角；旧网格规则仍可能把输入框或列表项拉伸。
- 对 Grid/Flex 子项设置 `min-width: 0`，防止文本和表单横向撑破容器。
- 表单从双列改为侧栏单列时，必须显式写 `grid-template-columns: minmax(0, 1fr)`。
- 修改后检查 `:first-child`、`:last-child` 等旧选择器是否因 DOM 结构变化命中了错误元素。
- 透明圆角面板内如果首尾区块有独立背景，必须给首尾区块设置与外框匹配的内圆角，避免矩形背景覆盖外框转角；如果 hover 背景由嵌套列表行自身绘制，末行也必须匹配底部内圆角。不要通过裁切整个面板解决，否则三点菜单等浮层会被截断。
- 论坛首页的版块导航不是圆角面板，其底部分隔线必须保持笔直；路由专属样式应显式清除共享首尾子元素规则带来的圆角。

曾经出现并已修复的问题：

- 权限组成员区继承旧 `display: grid` 后自动拉伸，造成搜索框巨大和列表被截断。
- 成员头像插入第一列后，旧的 `> span:first-child` 命中头像，姓名和用户名失去纵向布局。
- 权限组右侧 280px 侧栏继承双列表单规则，导致“用途说明”被挤成竖排并横向溢出。
- 论坛全局多层覆盖产生巨大空白和圆角残留，后改为路由独立样式。

## 文档与权限

- 文件夹树接口返回 `fileCount` 和 `children`。文件夹允许连同全部子文件夹、文档及其业务数据递归删除；前端必须先展示影响范围，再要求输入文件夹名称进行第二次确认，后端仍需校验递归删除标记和名称。上传附件保留在文件库并解除与被删除目录、文档的归属关系。
- 单独删除文档同样是永久删除，不使用归档状态；删除时清理文档直接权限，文档拥有的业务数据由数据库级联删除，上传附件继续保留在文件库。
- 文件夹和文件的权限入口位于各自三点菜单；文件编辑页的权限入口也位于更多菜单。
- 文档页右侧内容区同时显示当前目录的子文件夹和文档，文件夹固定排在文档之前；支持按最近更新、名称和类型切换排序，点击文件夹名称或“打开”进入该目录。
- 文档页右侧列表只显示“文件名”和“最近更新”两个信息列及行尾操作，不单独显示类型、状态列；草稿文档在名称前显示紧凑“草稿”标签。右上角“新建”使用下拉选单，提供“新建文件夹”和“创建文档”，界面不将文档称为文件。
- 文档页左侧位置栏同时显示文件夹与直属文件，文件夹分支默认展开并可逐级折叠；置顶文件夹与文件直接作为右侧文档表格表头下方的前几行显示，不使用独立置顶面板。置顶及排序仅作用于当前目录的直接子项，进入其他目录时显示该目录自身的置顶结果；所有有权访问该资源的用户可见置顶项，仅系统管理员可置顶、取消置顶并调整当前目录内多个置顶项的顺序。置顶行仍保留完整三点操作菜单。
- 权限弹窗必须携带明确的 `folder` 或 `file` 目标，不能隐式使用当前选中文件夹管理文件权限。
- 权限级别为 `owner`、`editor`、`lecturer`、`viewer`、`no_access`。
- 权限按钮的显示与隐藏只是体验优化，服务层必须检查有效权限。

## 已确认的产品决策

- 导航名称使用“论坛”，不使用“讨论”。
- 导航和页面名称使用“文档”，不使用“内容”或“课程内容”；“授课”统一称为“课件”，“素材库”统一称为“文件”。内部路由和代码标识暂不随界面名称调整。
- 创建在线测验不包含“选择测验范围”。
- 测验选项逐行添加，不让用户在大文本框中手动换行分隔。
- 课程演示根据一级、二级标题拆分页，支持方向键、空格、PageUp/PageDown、Home 和 End。
- “课件”是所有已登录用户可访问的独立板块；课件由文档段落快照与实时练习引用拼装，创建者和系统管理员可编辑。文档编辑页不提供直接课件展示入口。
- 登录页的演示账号由 `NEXT_PUBLIC_SHOW_DEMO_ACCOUNTS` 控制；该变量写入 Web 构建产物，生产容器修改后需要重新 build。
- AI 设置支持保存多个命名模型配置并显式切换当前配置；配置使用常见 OpenAI 兼容服务商预设，只有“其他兼容服务”允许填写 API 地址。回答温度由服务端固定为 `0.2`，不作为管理员可调参数；当前配置不能直接删除，必须先切换。
- AI 回答后的参考资料只显示紧凑的文档链接，不提供展开段落或引用块详情的交互。
- 加载历史 AI 对话时必须重新校验引用文档是否存在且当前用户仍可访问；已删除、已归档或因权限变化不可访问的引用统一显示“文件不存在”，且不提供跳转链接。
- 论坛主题列表、主帖作者和各级回复优先显示用户上传的头像；没有头像时统一降级为姓名首字母。
- 论坛支持匿名发帖和匿名回复，但数据库始终保留真实作者；API 必须对普通成员和管理员脱敏，只有最高管理员可以收到并查看匿名内容的真实作者身份。
- 论坛帖子标题、分类和主帖正文只有最高管理员可以修改；普通管理员仍可锁定、解锁和删除帖子，评论与嵌套回复发布后不提供编辑功能，评论作者和系统管理员仍可删除评论。
- 论坛帖子不使用归档状态；帖子作者或系统管理员删除帖子时永久删除整帖、全部回复和论坛专属图片。锁定仍用于暂停回复，但论坛列表不提供状态筛选栏目。
- 论坛主帖最多附带 9 张图片，评论和嵌套回复最多附带 3 张；浏览器上传前统一压缩为最长边不超过 1600px 的 WebP，服务端仍需根据首帖或回复分别校验真实文件头、尺寸、数量和容量。论坛图片作为帖子专属附件存储，不进入用户文件列表；匿名内容的图片响应不得暴露上传者或原始文件名。
- 论坛正文中的图片默认使用类似动态信息流的紧凑缩略图组，单图也不铺满正文；点击缩略图后在当前页面遮罩层查看大图，多图支持上一张、下一张、方向键和 Esc，不跳转新页面。
- 每个正常用户拥有独立个人主页，可自定义 Banner、头像、显示名和个人简介。业务界面展示作者时只显示可点击的显示名，不并排显示登录用户名；`@用户名` 只在个人主页和必要的账户管理界面显示。

## 数据与安全

- 系统角色为 `super_admin`、`admin`、`member`；普通用户的内容能力由权限组和资源授权决定。
- 后端服务层是权限安全边界。
- 非公开 API 统一通过全局活动用户守卫校验账号状态和 `sessionVersion`；停用账号、修改密码或管理员重置关键账号属性后，旧会话必须立即失效。
- 登录失败次数使用 Redis 计数，`TRUST_PROXY_HOPS` 必须与 API 前实际可信代理层数一致，避免直接信任客户端伪造的转发头。
- 上传内容不得以内联方式提供 SVG；只有经过文件头识别的 PNG、JPEG、GIF 和 WebP 可以内联，其他类型强制下载并使用 `nosniff`。安全图片响应使用 `Cross-Origin-Resource-Policy: same-site`，以支持本地 Web 与 API 不同端口的预览；下载型附件继续使用 `same-origin`。
- AI 服务商 API Key 使用 `AI_ENCRYPTION_KEY` 进行 AES-GCM 加密后存入数据库；生产部署必须保留该密钥，否则已有配置无法解密。
- AI 请求由 Redis 实施单用户窗口限流和并发限制；流式请求在浏览器断开后必须取消上游请求。Nginx 必须关闭 API 代理缓冲并保留长于模型请求的读取超时。
- AI 调用按次数限额：`User.aiCallCount` 累计用量，`User.aiCallLimit` 为成员例外（null 跟随 `AiSettings.defaultCallLimit`）。`AiService.consumeCallQuota` 是所有 AI 调用的统一消耗点（原子条件递增，超限抛 429），任何新增 AI 调用入口必须先经过它；消耗发生在配置校验之后，AI 未启用或配置不完整时不扣次数。
- 图片和附件区块写入 `assetId` 时，服务端必须校验附件工作区和重新引用权限；归档文件中的旧区块不得继续授予附件下载权限。
- 所有 schema 变更都应提交 Prisma migration，不使用 `db push`；测试数据库需要重建时使用 `pnpm db:reset`。
- Release 部署必须自动生成 `SESSION_SECRET`、数据库密码和 MinIO 凭据。HTTP IP 模式使用 `SESSION_COOKIE_SECURE=false`；配置 HTTPS 后改为 `true`。

## 验证清单

提交前按风险选择并记录：

```bash
pnpm format:check
pnpm typecheck
pnpm test
pnpm build
# 或一次执行
pnpm validate
```

UI 修改额外确认：

- 页面没有横向滚动或截断。
- 输入框、下拉框和列表项没有被父级 Grid 拉伸。
- 弹窗在小屏内可滚动且操作按钮可达。
- 空、有少量数据、有大量数据三种状态均合理。
- 三点菜单在视口边缘不会溢出。
